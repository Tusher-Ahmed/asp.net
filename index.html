ASP.NET Core — শূন্য থেকে ইউরোপের চাকরি পর্যন্ত

এই পেজটি একটি সম্পূর্ণ, ধারাবাহিক কারিকুলাম। প্রতিটি অধ্যায় আগেরটার ওপর দাঁড়িয়ে — তাই ওপর থেকে নিচে পড়ুন, প্রতিটি কোড নিজে টাইপ করে চালান। আর ৩+ বছরের অভিজ্ঞ ইঞ্জিনিয়ার হলে পর্ব ১–৫ দ্রুত ঝালিয়ে নিয়ে সোজা চলে যান পর্ব ৬-এ — runtime internals, distributed systems, observability, system design ও mid/senior লেভেলের ইন্টারভিউ-নেগোসিয়েশন সেখানে।

অধ্যায় ০১

ভূমিকা: ASP.NET Core কী এবং কেন শিখবেন

এই অধ্যায় শেষে জানবেন: ASP.NET Core কী, .NET-এর ভার্সনগুলো কীভাবে কাজ করে, আর কেন এটা ইউরোপে চাকরির জন্য অন্যতম সেরা টেক স্ট্যাক।

১.১ ASP.NET Core আসলে কী?

ASP.NET Core হলো Microsoft-এর তৈরি একটি ওপেন সোর্স, ক্রস-প্ল্যাটফর্ম ওয়েব ফ্রেমওয়ার্ক — অর্থাৎ এটা দিয়ে বানানো অ্যাপ Windows, Linux, macOS সব জায়গায় চলে। এটা দিয়ে আপনি বানাতে পারবেন:

  • Web API — মোবাইল অ্যাপ বা React/Angular ফ্রন্টএন্ডের জন্য ব্যাকএন্ড (ইউরোপে সবচেয়ে বেশি চাহিদা এটারই)
  • ওয়েবসাইট — MVC বা Razor Pages দিয়ে সার্ভার-রেন্ডার্ড সাইট
  • রিয়েল-টাইম অ্যাপ — SignalR দিয়ে চ্যাট, লাইভ নোটিফিকেশন
  • Microservices — বড় সিস্টেমকে ছোট ছোট সার্ভিসে ভাগ করা আর্কিটেকচার
  • Blazor — C# দিয়েই ইন্টারঅ্যাকটিভ ফ্রন্টএন্ড

ভাষা হিসেবে ব্যবহার হয় C# (উচ্চারণ: সি-শার্প) — একটি আধুনিক, টাইপ-সেফ, খুবই গোছানো ভাষা। পুরনো ASP.NET Framework (শুধু Windows-এ চলত) আর নতুন ASP.NET Core এক জিনিস নয় — আমরা শিখব শুধু নতুনটা।

১.২ .NET-এর ভার্সন বোঝা

Microsoft প্রতি বছর নভেম্বরে নতুন .NET রিলিজ করে। জোড় সংখ্যার ভার্সন হয় LTS (Long Term Support) — ৩ বছর সাপোর্ট পায়, কোম্পানিগুলো এগুলোই ব্যবহার করে।

.NET 9STSনভেম্বর ২০২৪১৮ মাস সাপোর্ট

.NET 8, 9, 10 — কোড প্রায় একই। এই কোর্সে যা শিখবেন তা সব ভার্সনে কাজ করবে। ইন্টারভিউতে বলবেন: "আমি .NET 8+ এ কাজ করি" — এটাই যথেষ্ট।

১.৩ কেন ইউরোপের জন্য .NET?

  • এন্টারপ্রাইজের ভাষা: জার্মানি, নেদারল্যান্ডস, নর্ডিক দেশগুলোর ব্যাংক, ইনস্যুরেন্স, লজিস্টিকস, ম্যানুফ্যাকচারিং কোম্পানির বিশাল অংশ .NET-এ চলে। এসব কোম্পানি স্থিতিশীল, বেতন ভালো, ভিসা স্পনসর করে।
  • প্রতিযোগিতা তুলনামূলক কম: JavaScript/Python-এ প্রার্থীর ভিড় বেশি; অভিজ্ঞ .NET ডেভেলপারের সাপ্লাই কম, চাহিদা ধারাবাহিক।
  • একটা স্কিলে অনেক দরজা: C# জানলে ব্যাকএন্ড, ক্লাউড (Azure), ডেস্কটপ, এমনকি গেম (Unity) — সব খোলা।
  • Azure-এর সাথে জুটি: ইউরোপের অনেক কোম্পানি Microsoft Azure ক্লাউড ব্যবহার করে; .NET + Azure জানা প্রার্থী তারা সবসময় খোঁজে।

১.৪ এই কোর্স কীভাবে পড়বেন

  1. ক্রমানুসারে পড়ুন। প্রতিটি অধ্যায় আগের অধ্যায়ের জ্ঞান ধরে নেয়।
  2. প্রতিটি কোড নিজে টাইপ করুন। কপি-পেস্ট করলে হাত শিখবে না। এরর আসবে — সেটাই শেখা।
  3. প্রতি পর্ব শেষে ছোট প্রজেক্ট করুন। অধ্যায় ২৫-এ প্রজেক্ট আইডিয়া দেওয়া আছে।
  4. ইংরেজি টার্মগুলো ইংরেজিতেই মনে রাখুন। ইন্টারভিউ ইংরেজিতে হবে — তাই আমি ব্যাখ্যা বাংলায় দিলেও টেকনিক্যাল শব্দ ইংরেজিতে রেখেছি।

২.৫ Git — প্রথম দিন থেকেই

ইউরোপের প্রতিটি টিম Git-এ কাজ করে — এটা ঐচ্ছিক স্কিল নয়, ভাষার মতোই ভিত্তি। আজকের HelloBangla প্রজেক্ট থেকেই প্রতিটি অনুশীলন Git-এ রাখুন:

টিমের স্বাভাবিক প্রবাহ (যেটা ইন্টারভিউতেও জিজ্ঞেস করে): main ব্রাঞ্চে সরাসরি কেউ লেখে না → ফিচার ব্রাঞ্চ → push → Pull Request → সহকর্মীর code review → CI টেস্ট পাশ → merge। আরও শিখে রাখুন: git log, git diff, merge conflict সমাধান, .gitignore (bin/, obj/ কখনো commit নয় — dotnet new gitignore কমান্ডেই রেডিমেড পাবেন)।

অধ্যায় ০৩

C# — ওয়েব ডেভেলপমেন্টের জন্য জরুরি বেসিক

ASP.NET Core শেখার আগে C#-এর যে অংশগুলো অবশ্যই লাগবে — টাইপ, ক্লাস, ইন্টারফেস, async/await, LINQ — সব এক জায়গায়।

গুরুত্বপূর্ণ

C# একদম নতুন হলে এই অধ্যায়টা ২–৩ সপ্তাহ সময় নিয়ে প্র্যাকটিস করুন। ফ্রেমওয়ার্ক শেখা সহজ, কিন্তু ভাষার ভিত দুর্বল হলে সামনে সব কঠিন লাগবে। ইন্টারভিউতেও C# ফান্ডামেন্টাল প্রশ্ন আসবেই।

৩.১ ভ্যারিয়েবল ও টাইপ

C# একটি statically typed ভাষা — প্রতিটি ভ্যারিয়েবলের টাইপ কম্পাইলের সময়ই নির্ধারিত। এতে ভুল আগেই ধরা পড়ে।

৩.২ Nullable — null থেকে বাঁচার ব্যবস্থা

আধুনিক C#-এ কোনো ভ্যারিয়েবল null হতে পারবে কি না তা টাইপেই বলে দিতে হয়। টাইপের পরে ? মানে "এটা null হতে পারে"।

৪.৫ HTTP প্রাইমার — যে ভাষায় ওয়েব কথা বলে

ওয়েব ডেভেলপার হিসেবে আপনার প্রতিটা কাজের নিচে HTTP। একটা রিকোয়েস্ট-রেসপন্স জোড়া আসলে দেখতে এমন:

  • Method-এর অর্থ: GET (পড়া, body নেই), POST (তৈরি), PUT (পুরো প্রতিস্থাপন), PATCH (আংশিক বদল), DELETE (মুছে ফেলা)
  • Idempotent মানে: একই রিকোয়েস্ট বারবার পাঠালেও ফল একই — GET, PUT, DELETE idempotent; POST নয় (দুবার পাঠালে দুটো অর্ডার!)। নেটওয়ার্ক retry-এর যুগে এই ধারণাটা ইন্টারভিউয়ের প্রিয় প্রশ্ন।
  • স্ট্যাটাস কোডের পরিবার: 2xx সফল · 3xx redirect · 4xx ক্লায়েন্টের ভুল · 5xx সার্ভারের ভুল — "৪ মানে তুমি, ৫ মানে আমি"
  • Header: মেটাডেটা — কে পাঠাচ্ছে, কী ফরম্যাটে, ক্যাশ করা যাবে কি না। ব্রাউজারের DevTools → Network ট্যাবে যেকোনো সাইটের আসল ট্রাফিক দেখে অভ্যাস করুন।

৫.১ Pipeline কী?

৫.৫ Filter — endpoint-এর গায়ের কাছের স্তর

Middleware চলে সব রিকোয়েস্টে; filter চলে নির্দিষ্ট endpoint/controller-এ, এবং model binding-এর পরে — তাই filter প্যারামিটার-ভ্যালিডেশনের মতো কাজ পারে যা middleware পারে না। এই পার্থক্যটা (roadmap.sh-এও আলাদা টপিক) ইন্টারভিউতে আসে।

MVC-তে filter-এর ধরন: Authorization → Resource → Action → Exception → Result — এই ক্রমে চলে। মনে রাখার সূত্র: cross-cutting জিনিস সব রিকোয়েস্টে লাগলে middleware, নির্দিষ্ট কিছু endpoint-এ লাগলে filter।

১৬.২ API-র জন্য JWT — কীভাবে কাজ করে

  1. ইউজার email/password দিয়ে POST /login করে

২১.৪ GDPR — ইউরোপে কাজের বাড়তি বাস্তবতা

ইউরোপে ব্যক্তিগত ডেটা (নাম, ইমেইল, IP পর্যন্ত!) আইন দিয়ে সুরক্ষিত। ডেভেলপার হিসেবে যা জানা ভালো: যত কম ব্যক্তিগত ডেটা রাখা যায় তত ভালো (data minimization), ইউজার চাইলে ডেটা মুছে দেওয়ার ব্যবস্থা রাখতে হয় (right to erasure), আর ব্যক্তিগত ডেটা লগে লেখা যায় না। ইন্টারভিউতে GDPR-সচেতনতা দেখালে ভালো ছাপ পড়ে।

২৩.১ SOLID — পাঁচ নীতি, সহজ ভাষায়

২৩.২ Clean Architecture — স্তরে স্তরে

মূল কথা: বিজনেস লজিক কেন্দ্রে, বাইরের জিনিস (DB, ওয়েব) প্রান্তে — নির্ভরতার তীর সবসময় ভেতরের দিকে। প্রচলিত চার প্রজেক্টের কাঠামো:

লাভ: ডাটাবেস বদলালে Domain/Application ছোঁয়া লাগে না; বিজনেস লজিক DB ছাড়াই টেস্ট করা যায়। জুনিয়র প্রজেক্টে এত ভাগ overkill হতে পারে — তবে ধারণাটা ইন্টারভিউতে ব্যাখ্যা করতে পারা জরুরি।

২৩.৩ Repository ও CQRS — দুটো বহুশ্রুত প্যাটার্ন

  • Repository pattern: ডেটা-অ্যাক্সেস একটা interface-এর পেছনে (IProductRepository.GetAsync(id)) — বিজনেস লজিক জানে না নিচে EF নাকি Dapper। সমালোচনাও জানুন: EF-এর DbSet নিজেই অনেকটা repository, তাই অনেক টিম বাড়তি স্তর বানায় না।
  • CQRS: Command (লেখা) আর Query (পড়া) আলাদা পথ। .NET জগতে খুবই প্রচলিত MediatR লাইব্রেরিসহ: প্রতিটি অপারেশন = একটা ছোট Handler ক্লাস। বড় কোডবেস পরিষ্কার থাকে।

২৩.৪ Microservices — ধারণাটুকু

এক বিশাল অ্যাপ (monolith)-এর বদলে ছোট ছোট স্বাধীন সার্ভিস — Orders, Payments, Inventory — প্রত্যেকের নিজের ডাটাবেস, নিজের ডিপ্লয়মেন্ট। যোগাযোগ হয় HTTP/gRPC (সরাসরি) বা message broker দিয়ে (RabbitMQ, Kafka — "OrderPlaced" ইভেন্ট ছাড়লে যার দরকার সে শোনে)।

জুনিয়র হিসেবে সঠিক অবস্থান

ইন্টারভিউতে বলুন: "Monolith দিয়ে শুরু করাই ভালো; টিম ও ডোমেইন বড় হলে ভাগ করা যায়। Microservices আনে distributed system-এর জটিলতা — network failure, eventual consistency, observability।" এই সংযত উত্তর মুখস্থ buzzword-এর চেয়ে অনেক বেশি নম্বর পায়।

২৬.১ Memory: stack, heap, boxing

  • Value type (int, struct, enum) — মান সরাসরি কপি হয়; লোকাল ভ্যারিয়েবল হলে সাধারণত stack-এ থাকে
  • Reference type (class, string, array) — অবজেক্ট heap-এ, ভ্যারিয়েবলে থাকে রেফারেন্স
  • Boxing: value type-কে object-এ ভরলে heap-এ নতুন অবজেক্ট তৈরি হয় — লুপের ভেতরে হলে GC চাপে পড়ে। generic (List<int>) boxing এড়ায় — এটাই generic-এর জন্মের অন্যতম কারণ।

২৬.২ Garbage Collector — যা জানা must

  • Generational: Gen0 (নতুন, ঘনঘন collect), Gen1, Gen2 (দীর্ঘজীবী, দামি collect)। "বেশিরভাগ অবজেক্ট অল্পদিন বাঁচে" — এই অনুমানেই ডিজাইন।
  • LOH (Large Object Heap): ~85KB+ অবজেক্ট আলাদা জায়গায়, compact হয় না সহজে — বড় array বারবার বানালে fragmentation। সমাধান: ArrayPool<T>.Shared থেকে ভাড়া নিন।
  • Server GC vs Workstation GC: ASP.NET Core ডিফল্টে Server GC — প্রতি কোরে আলাদা heap, throughput বেশি।
  • IDisposable ≠ GC: GC মেমোরি সামলায়; ফাইল হ্যান্ডেল, সকেট, DB connection-এর মতো unmanaged রিসোর্স ছাড়তে হয় Dispose/using দিয়ে — deterministic ভাবে।
ইন্টারভিউ প্রশ্ন

"finalizer আর Dispose-এর পার্থক্য?" — Dispose আপনি ডাকেন (deterministic), finalizer GC ডাকে (কখন কেউ জানে না, দামি)। সঠিক প্যাটার্ন: Dispose pattern + GC.SuppressFinalize; নিজের কোডে finalizer প্রায় কখনোই লাগে না — SafeHandle ব্যবহার করুন।

২৬.৩ Threading ও async — গভীরে

  • ConfigureAwait(false): লাইব্রেরি কোডে context-এ ফেরার দরকার নেই বলে লেখা হয়; ASP.NET Core-এ SynchronizationContext নেই, তাই অ্যাপ কোডে এটা কার্যত no-op — এই সূক্ষ্মতা বললে সিনিয়র শোনায়।
  • ValueTask: প্রায়ই synchronously শেষ হওয়া hot path-এ Task-এর allocation বাঁচায়; দুবার await করা যায় না — জানুন, কিন্তু অকারণে ব্যবহার করবেন না।
  • Channel<T>: producer/consumer-এর আধুনিক সমাধান (অধ্যায় ১৯-এ দেখেছেন) — BlockingCollection-এর async উত্তরসূরি।

২৭.১ Optimistic concurrency — দুজন একসাথে সেভ করলে?

দুই ইউজার একই রেকর্ড এডিট করছে — শেষজনের লেখা চুপচাপ আগেরটা মুছে দেবে (lost update)। সমাধান: ভার্সন কলাম, mismatch হলে exception:

Pessimistic (আগে থেকে lock — SELECT ... FOR UPDATE) লাগে টিকিট বুকিং-এর মতো hot row-তে; বাকি সব জায়গায় optimistic-ই ডিফল্ট। এই তুলনাটা সিস্টেম-ডিজাইন ইন্টারভিউতেও আসে।

২৭.৫ Outbox pattern — DB আর message broker-এর সমন্বয়

"অর্ডার সেভ করলাম + ইভেন্ট পাঠালাম" — মাঝখানে অ্যাপ মরলে? DB-তে আছে কিন্তু ইভেন্ট যায়নি (বা উল্টো)। সমাধান: ইভেন্টও একই transaction-এ একটা Outbox টেবিলে লিখুন; আলাদা background worker সেখান থেকে পড়ে broker-এ পাঠায়:

কেন এটা জরুরি

Outbox + idempotent consumer (পরের অধ্যায়) — এই জুটি distributed system ইন্টারভিউয়ের অন্যতম প্রিয় আলোচনা। MassTransit/CAP-এর মতো লাইব্রেরিতে বিল্ট-ইন আছে, কিন্তু নিজে ব্যাখ্যা করতে পারতে হবে।

  • Retry + Dead Letter Queue: consumer fail করলে কয়েকবার retry (backoff-সহ), তারপর মেসেজ DLQ-তে — সেখানে alert + ম্যানুয়াল তদন্ত। "poison message" পুরো queue আটকাবে না।
  • Saga: কয়েক সার্ভিস জুড়ে লেনদেন (অর্ডার→পেমেন্ট→শিপিং) — distributed transaction নেই, বদলে প্রতি ধাপের compensating action (পেমেন্ট fail → অর্ডার cancel)। Choreography (ইভেন্ট শুনে শুনে) vs Orchestration (এক সমন্বয়ক state machine — MassTransit-এ আছে)।
  • Kafka কখন: event stream/replay/বিপুল throughput দরকার হলে; সাধারণ task queue-তে RabbitMQ-ই সহজ ও যথেষ্ট।

২৮.২ gRPC — সার্ভিস-টু-সার্ভিস দ্রুত যোগাযোগ

তৃতীয় বিকল্পটাও চিনে রাখুন: GraphQL (.NET-এ HotChocolate লাইব্রেরি) — ক্লায়েন্ট নিজেই বলে দেয় কোন কোন ফিল্ড চাই; অনেক ভিন্ন ক্লায়েন্টের (mobile/web) ভিন্ন ডেটা-চাহিদা থাকলে কাজে দেয়। ইউরোপের কিছু প্রোডাক্ট কোম্পানিতে আছে — নাম ও কখন-কেন জানা থাকলেই ইন্টারভিউতে যথেষ্ট।

২৮.৩ Resilience — নেটওয়ার্ক fail করবেই

২৮.৪ DDD — ধারণাগুলো যেগুলো টিমে রোজ শুনবেন

  • Ubiquitous language: কোডের নাম = ব্যবসার ভাষা। ডোমেইন এক্সপার্ট "Booking" বললে ক্লাসের নাম ReservationRecordDto নয়, Booking।
  • Bounded context: "Product" শব্দটা Sales আর Inventory-তে ভিন্ন জিনিস — প্রতিটি context-এর নিজের মডেল, জোর করে এক টেবিলে নয়। Microservice ভাগের প্রাকৃতিক সীমানাও এটাই।
  • Aggregate: একসাথে consistent থাকতে হয় এমন entity-গুচ্ছ (Order + OrderLines), বাইরে থেকে ঢোকার একটাই দরজা — aggregate root। এক transaction = এক aggregate, এই নিয়মটা মনে রাখুন।
  • Domain event: "OrderPlaced" — ডোমেইনে ঘটা ঘটনা, যেটা outbox হয়ে integration event-এ রূপ নেয়।
Modular monolith — সৎ উত্তর

২০২৪+ ইন্ডাস্ট্রির স্পষ্ট ঝোঁক: আগে modular monolith (এক ডিপ্লয়মেন্ট, ভেতরে কঠোরভাবে আলাদা মডিউল — নিজ নিজ DbContext, যোগাযোগ interface/ইভেন্টে), সত্যিকারের প্রয়োজনে মডিউল খুলে আলাদা সার্ভিস। ইন্টারভিউতে এই অবস্থান নিলে অভিজ্ঞতা ঝরে পড়ে। .NET Aspire-ও চিনে রাখুন — লোকালে multi-service অ্যাপ চালানো/অবজার্ভ করার নতুন অফিসিয়াল টুলিং।

অধ্যায় ২৯ · অভিজ্ঞ লেভেল

Observability — OpenTelemetry হাতে-কলমে

"প্রোডাকশনে ধীর কেন?" — এই প্রশ্নের উত্তর দিতে পারা মানুষটাই টিমের সবচেয়ে দামি। Traces, metrics, logs — সেটআপসহ।

২৯.১ তিন স্তম্ভ, এক স্ট্যান্ডার্ড

  • Logs — কী ঘটেছে (অধ্যায় ৯)
  • Metrics — সংখ্যায় স্বাস্থ্য: RPS, latency p95, error rate, queue depth
  • Traces — এক রিকোয়েস্টের পুরো যাত্রা: gateway → API → DB → broker → অন্য সার্ভিস, প্রতিটি ধাপের সময়সহ

OpenTelemetry (OTel) তিনটারই vendor-নিরপেক্ষ স্ট্যান্ডার্ড — একবার instrument করলে backend যা খুশি: Jaeger/Tempo (trace), Prometheus/Grafana (metrics), Loki/Seq (logs), বা Azure Application Insights, Datadog।

২৯.২ সেটআপ

জাদুটা হলো context propagation: traceparent header আপনাআপনি সার্ভিস থেকে সার্ভিসে যায় — Grafana/Jaeger-এ এক ক্লিকে দেখবেন কোন রিকোয়েস্টের ৮০০ms-এর ৭২০ms গেছে কোন SQL-এ।

২৯.৩ SLO-মাইন্ডসেট — সিনিয়র শোনানোর শব্দভান্ডার

  • SLI (indicator): মাপা জিনিস — "p95 latency", "success rate"
  • SLO (objective): লক্ষ্য — "৩০ দিনে ৯৯.৯% রিকোয়েস্ট < 300ms"
  • Error budget: ০.১% fail করার "অনুমতি" — budget শেষ হলে ফিচার থামিয়ে stability-তে কাজ; ব্যবসা আর ইঞ্জিনিয়ারিং-এর দর-কষাকষির ভাষা
  • Alert নিয়ম: symptom-এ alert করুন (error rate বাড়ছে), cause-এ নয় (CPU 80%); রাতে যে alert-এ কিছু করার নেই সেটা alert নয়, spam
ইন্টারভিউ গোল্ড

"প্রোডাকশন incident সামলেছেন কখনো?" প্রশ্নে STAR গল্প রেডি রাখুন: alert পেলাম → trace/লগে সংকীর্ণ করলাম → mitigate (rollback/feature flag) → post-mortem-এ root cause + প্রতিরোধ। blameless post-mortem শব্দটা ব্যবহার করুন — ইউরোপীয় ইঞ্জিনিয়ারিং কালচারে এটা মূল্যবোধের সংকেত।

অধ্যায় ৩০ · অভিজ্ঞ লেভেল

System Design ইন্টারভিউ — কাঠামো ও অনুশীলন

Mid-level থেকে ওপরে উঠলেই লুপে system design রাউন্ড থাকবে। এখানে চলার কাঠামো + একটা পূর্ণ উদাহরণ।

৩০.১ ৪৫ মিনিটের খেলার নিয়ম

  1. Requirements (৫ মি): functional ("কী করবে") + non-functional ("কত ইউজার? latency? consistency নাকি availability জরুরি?")। প্রশ্ন না করে ডিজাইন শুরু করাই এক নম্বর ভুল।
  2. Estimation (৫ মি): মোটা দাগে — DAU × request/user ÷ 86400 ≈ RPS; storage = রেকর্ড × সাইজ × বছর। নিখুঁত নয়, যুক্তিসঙ্গত।
  3. API + data model (১০ মি): মূল endpoint, টেবিল/কী, কোন DB কেন
  4. High-level ডায়াগ্রাম (১০ মি): client → LB → API → cache/DB/queue — বাক্স আর তীর
  5. Deep dive + trade-off (১৫ মি): ইন্টারভিউয়ার যেদিকে টানে — scaling, failure, consistency। প্রতি সিদ্ধান্তে "বিকল্প ছিল X, নিলাম Y কারণ…"

  • অভিজ্ঞদের ক্ষেত্রে পোর্টফোলিও প্রজেক্টের চেয়ে কাজের গল্প বেশি ওজন রাখে; GitHub তবু সবুজ রাখুন — এই কোর্সের advanced প্রজেক্টগুলো (outbox, OTel-সহ) দেখানোর মতো জিনিস
  • প্রতিটি আবেদনের আগে job description-এর কীওয়ার্ড CV-তে মিলিয়ে নিন (ATS ফিল্টার)

৩১.৩ অভিজ্ঞদের ইন্টারভিউ লুপ — কী বদলায়

  • System design রাউন্ড যোগ হয় (অধ্যায় ৩০) — এটাই mid/senior-এর ছাঁকনি
  • Behavioral গভীর হয়: "সবচেয়ে কঠিন টেকনিক্যাল সিদ্ধান্ত?", "সহকর্মীর সাথে দ্বিমত হলে?", "একটা ব্যর্থতার গল্প" — ৬-৮টা STAR গল্প লিখে রাখুন: migration, incident, দ্বিমত→সমাধান, mentoring, deadline trade-off
  • Code review রাউন্ড অনেক কোম্পানিতে: খারাপ PR দিয়ে বলবে review করুন — bug, সিকিউরিটি, নামকরণ, টেস্টের অভাব ধরুন; সুরটা গঠনমূলক রাখুন, সেটাও মাপা হচ্ছে
  • আপনিও ইন্টারভিউ নিন: টিমের engineering culture প্রশ্ন করুন — CI/CD কেমন, on-call rotation, tech debt-এর প্রতি মনোভাব। ভালো প্রশ্ন নিজেই সিনিয়রিটির প্রমাণ।

৩২.৪ Integration patterns — অন্য সিস্টেমের সাথে জোড়া

  • Sync (request/response): সহজ, কিন্তু ডাকা সার্ভিস ধীর/মৃত হলে আপনিও — তাই সবসময় timeout + retry + circuit breaker (অধ্যায় ২৮.৩)।
  • Async (event/queue): decoupled ও স্থিতিস্থাপক (অধ্যায় ২৮.১) — কিন্তু eventual consistency মেনে নিতে হয়।
  • Webhook: "কিছু ঘটলে আমি তোমাকে ডাকব" — পেমেন্ট গেটওয়ে এভাবেই ফল জানায়। রিসিভার বানানোর নিয়ম: সিগনেচার যাচাই করুন (HMAC header), দ্রুত 200 দিন (কাজ queue-তে ফেলে), duplicate সামলান (তারা retry করে)।
  • Polling বনাম webhook বনাম SSE/WebSocket: মাঝে মাঝে চেক চলবে → polling; সোর্স push দিতে পারে → webhook (সার্ভার-টু-সার্ভার) বা SignalR/SSE (ব্রাউজারে)।
  • Anti-corruption layer: বাইরের API-র অগোছালো মডেল সরাসরি নিজের ডোমেইনে ঢোকাবেন না — মাঝে একটা অনুবাদক ক্লাস (adapter) রাখুন; বাইরেরটা বদলালে শুধু adapter বদলায়। DDD-র (অধ্যায় ২৮.৪) বাস্তব প্রয়োগ।
  • BFF (Backend-for-Frontend): মোবাইল আর ওয়েবের চাহিদা আলাদা হলে প্রত্যেকের জন্য পাতলা নিজস্ব API-স্তর — ক্লায়েন্টপ্রতি দর্জি-করা রেসপন্স, ভেতরে একই সার্ভিস।

৩২.৭ Richardson Maturity Model — "আপনার API কতটা RESTful?"

  • Level 0: এক URL-এ সব POST (RPC-স্টাইল) · Level 1: আলাদা resource URL · Level 2: + সঠিক HTTP method ও status code — ইন্ডাস্ট্রির বাস্তব মান এটাই · Level 3: + HATEOAS — রেসপন্সের ভেতরে পরের ধাপের লিংক ("links": [{"rel":"cancel", "href":"/orders/42/cancel"}])।
  • সৎ উত্তর: "আমরা Level 2 বানাই; HATEOAS তত্ত্বে সুন্দর, বাস্তবে ক্লায়েন্টরা কমই ব্যবহার করে" — এই nuance-টাই ইন্টারভিউয়ার শুনতে চায়।
  • Contract-first বনাম code-first: আগে OpenAPI স্পেক লিখে টিমগুলো একমত হওয়া (বড় সংস্থা/একাধিক টিম) বনাম কোড থেকে স্পেক জেনারেট (ছোট টিম, দ্রুত) — দুটোরই নাম জানুন।

৩৩.২ Schema design — সিদ্ধান্তগুলো

  • Normalization (3NF পর্যন্ত): প্রতিটি তথ্য এক জায়গায় — customer-এর ঠিকানা orders টেবিলে কপি নয়, customers-এ একবার। আপডেট-অসঙ্গতি (anomaly) ঠেকায়।
  • ইচ্ছাকৃত denormalization: read-ভারী জায়গায় হিসাব করা মান (যেমন orders.total_amount) রেখে দেওয়া বৈধ কৌশল — শর্ত: কে/কখন আপডেট রাখবে তা পরিষ্কার। "নর্মালাইজ করে শুরু, মেপে denormalize" — ইন্টারভিউ-বাক্য।
  • ডেটা টাইপ কৃপণের মতো: টাকা DECIMAL(10,2) (কখনোই FLOAT নয়!), স্ট্যাটাস TINYINT/enum, সময় DATETIME(6) UTC-তে, ID-তে BIGINT AUTO_INCREMENT বা UUIDv7 (এলোমেলো UUIDv4 primary key-তে InnoDB-র B-tree ছত্রখান করে — সূক্ষ্ম কিন্তু দামি জ্ঞান)।
  • FK constraint রাখুন — "অ্যাপে সামলাবো" বলে বাদ দিলে একদিন এতিম সারি (orphan rows) পাবেনই।

৩৩.৩ Index — কীভাবে ভাবতে হয়

  • Composite index-এ ক্রমই সব: (customer_id, created_at) index কাজ করে "এই customer-এর অর্ডার তারিখক্রমে" কোয়েরিতে; কিন্তু শুধু created_at দিয়ে খুঁজলে কাজে লাগে না — বাঁ দিক থেকে ব্যবহার হয় (leftmost prefix rule)।
  • Covering index: কোয়েরির সব কলাম index-এই থাকলে টেবিল ছোঁয়াই লাগে না — EXPLAIN-এ Using index দেখলে বুঝবেন জিতেছেন।
  • Index অকেজো হয়ে যায় যখন: কলামে ফাংশন (WHERE YEAR(created_at)=2026 ❌ → রেঞ্জ দিন ✅), আগে-% যুক্ত LIKE ('%abc'), টাইপ-অমিল (string কলামে number তুলনা)।
  • দাম আছে: প্রতিটি index মানে প্রতি INSERT/UPDATE-এ বাড়তি লেখা — "সব কলামে index" সমাধান নয়, রোগ।

৩৩.৪ ধীর কোয়েরি ধরা ও সারানো — কর্মপ্রবাহ

সারানোর সাধারণ ক্রম: সঠিক index যোগ → SELECT * বাদ দিয়ে দরকারি কলাম (EF-তে Select() projection — অধ্যায় ১৪) → offset-এর বদলে keyset pagination (৩২.৩) → তবু ভারী হলে ক্যাশ (অধ্যায় ১৮) বা read replica (অধ্যায় ৩০)।

৩৩.৫ ক্লাসিক SQL ইন্টারভিউ কোয়েরি — হাতে লিখতে পারতে হবে

  • WHERE বনাম HAVING: WHERE ছাঁকে গ্রুপ করার আগে (সারি ধরে), HAVING গ্রুপ করার পরে (aggregate-এর ওপর)।
  • JOIN-এর প্রকার এক লাইনে: INNER = দুই পাশেই মিল আছে এমন; LEFT = বাঁয়ের সব + ডানে মিললে; RIGHT = উল্টোটা; CROSS = সব×সব। "LEFT JOIN + IS NULL" = "নেই" খোঁজার অস্ত্র।
  • DELETE বনাম TRUNCATE: DELETE সারি ধরে ধরে (WHERE চলে, rollback হয়, ধীর); TRUNCATE পুরো টেবিল রিসেট (দ্রুত, DDL, auto-increment-ও রিসেট)।
  • VARCHAR বনাম TEXT: ছোট-নির্দিষ্ট দৈর্ঘ্য ও index দরকার হলে VARCHAR; বড়/সীমাহীন কনটেন্টে TEXT (index-এ prefix লাগে)।

৩৩.৬ Clustered index, locking ও deadlock

  • InnoDB-তে primary key-ই clustered index — টেবিলের সারিগুলো ভৌতভাবে PK-ক্রমে সাজানো থাকে; প্রতিটি secondary index-এর পাতায় থাকে PK-র কপি (তাই মোটা PK = সব index মোটা)। এ কারণেই: PK ছোট ও ক্রমবর্ধমান রাখুন (৩৩.২-এর UUIDv4 সতর্কতার আসল কারণ এটাই) — এই ব্যাখ্যা দিতে পারা = শক্ত প্রার্থী।
  • Deadlock: দুই transaction উল্টো ক্রমে দুটো সারি lock করতে গেলে — MySQL একজনকে মেরে দেয় (Deadlock found; try restarting transaction)। প্রতিরোধ: সব জায়গায় একই ক্রমে সারি ধরুন, transaction ছোট রাখুন; প্রতিকার: ধরা পড়লে অ্যাপে retry (এটা স্বাভাবিক ঘটনা, বাগ নয়) — EF-এ execution strategy বা Polly দিয়ে।
  • এখনই কী চলছে দেখা: SHOW PROCESSLIST; (কোন কোয়েরি কতক্ষণ ঝুলছে), SHOW ENGINE INNODB STATUS; (শেষ deadlock-এর ময়নাতদন্ত), আটকে থাকা কোয়েরি KILL <id>;
  • Read/write split: লেখা primary-তে, পড়া replica-য় — কিন্তু replication lag মানে "এইমাত্র লিখে সাথে সাথে পড়লে" পুরনো ডেটা পেতে পারেন; সমাধান: নিজের-লেখা-নিজে-পড়া (read-your-writes) কোয়েরিগুলো primary-তে পাঠানো। (অধ্যায় ৩০-এর টেবিলের সাথে মেলে)

Performance, Scalability ও Maintainability Engineering

দ্রুত অ্যাপ, বাড়তে-পারা সিস্টেম, আর দু-বছর পরেও বদলানো-যায় এমন কোডবেস — তিনটা একসাথে চালানোর ইঞ্জিনিয়ারিং।

৩৪.১ Performance: মাপো → খোঁজো → সারাও → আবার মাপো

  1. Baseline মাপুন: load-testing টুল দিয়ে (k6 জনপ্রিয়, সহজ) — এখনকার RPS আর p95 কত?
  2. Bottleneck খুঁজুন: trace (অধ্যায় ২৯) বলবে সময় কোথায় যাচ্ছে — ৯০% ক্ষেত্রে উত্তর: ডাটাবেস।
  3. সারান — সস্তা থেকে দামি ক্রমে: index/কোয়েরি (৩৩.৪) → N+1/AsNoTracking (১৪) → ক্যাশ (১৮) → ধীর কাজ queue-তে (১৯) → তবেই hardware/architecture।
  4. আবার মাপুন — সংখ্যা ছাড়া "দ্রুত হয়েছে" দাবি ইঞ্জিনিয়ারিং নয়।

অ্যাপ-সাইডে দুটো নিত্য অস্ত্র: dotnet-counters monitor (লাইভ RPS, GC, থ্রেডপুল queue) আর response compression + System.Text.Json source-gen serialization — সস্তায় বড় লাভ।

৩৪.২ Scalability: বড় হওয়ার নকশা

  • Vertical (বড় মেশিন) — সহজ কিন্তু ছাদ আছে; Horizontal (বেশি মেশিন) — ছাদ নেই, কিন্তু শর্ত একটাই: অ্যাপ stateless হতে হবে। Session/ক্যাশ Redis-এ (১৮), ফাইল object storage-এ (১৩.৪), SignalR-এ Redis backplane — তাহলেই load balancer-এর পেছনে যত খুশি কপি।
  • ডাটাবেসই প্রথম আটকায়: ক্রম মনে রাখুন — connection pooling → read replica (পড়া আলাদা) → ক্যাশ-স্তর → sharding (একদম শেষ অস্ত্র — জটিলতা বিশাল)।
  • Queue = ধাক্কা-শোষক: traffic spike-এ সব কাজ সাথে সাথে করার দরকার নেই — accept করুন, queue-তে ফেলুন (১৯, ২৮), শান্তভাবে process করুন। Black Friday এভাবেই বাঁচে।
  • Backpressure ও graceful degradation: সিস্টেম ভরে গেলে rate limit (১৮.৪) + অগুরুত্বপূর্ণ ফিচার সাময়িক বন্ধ (recommendation বন্ধ, checkout চালু) — পুরো ডোবার চেয়ে আংশিক সেবা ভালো।

৩৪.৩ Maintainability: দু-বছর পরের সহকর্মীর জন্য

কোড একবার লেখা হয়, পঞ্চাশবার পড়া হয়। ইউরোপের টিমগুলো এই স্কিলটা code review-তেই মেপে নেয়:

  • Analyzer দিয়ে মান জোর করুন: .editorconfig + Roslyn analyzer + TreatWarningsAsErrors — স্টাইল-তর্ক মানুষে নয়, টুলে মিটুক। dotnet format CI-তে।
  • নামকরণই অর্ধেক ডকুমেন্টেশন: ProcessData() নয়, RecalculateMonthlyInvoiceTotals(); boolean-এ প্রশ্নবোধক নাম (isEligibleForRefund)।
  • ছোট PR, বর্ণনাসহ: ৩০০ লাইনের নিচে রাখুন; বর্ণনায় কী নয় (diff-ই বলে), কেন লিখুন। ভালো commit message-ও তাই।
  • Tech debt ব্যবস্থাপনা: দেখা মাত্র // TODO নয় — ticket খুলুন, প্রতি sprint-এ ২০% সময় debt-এ — এই ভাষায় ইন্টারভিউতে কথা বলুন।
  • Boy-scout rule: যে ফাইলে কাজ করছেন, একটু পরিষ্কার করে রেখে আসুন — refactoring আলাদা প্রকল্প নয়, অভ্যাস।

৩৪.৫ Horizontal scaling-এর ফাঁদগুলো

  • Sticky session-এ ভরসা নয়: "এই ইউজার সবসময় সার্ভার-৩-এ" — deploy/অটোস্কেলে ভাঙে। State বাইরে রাখুন (Redis), যেকোনো instance যেকোনো রিকোয়েস্ট নিক।
  • ইন-প্রসেস জিনিস আর কাজ করে না: lock এক প্রসেসেই সীমাবদ্ধ — একাধিক instance-এ লাগবে distributed lock (Redis SET NX EX) বা আরও ভালো: DB-র unique constraint/optimistic concurrency-তে সমস্যাটাই এড়ানো। IMemoryCache-ও instance-প্রতি আলাদা (তাই ১৮.২-এর HybridCache)।
  • Scheduled job × N instance = N বার চলা: BackgroundService সব কপিতে চলে! সমাধান: leader election / distributed lock / job আলাদা single-instance সার্ভিসে (Hangfire-এর নিজস্ব সমাধান আছে)।
  • 12-Factor App নামটা বলুন: config env-এ (৮), stateless process, লগ stdout-এ, dev/prod সমতা — cloud-native চাকরির শব্দভান্ডার।

৩৪.৬ Legacy কোডবেসে কাজ — "এটা আমরা রোজ করি"

আন্তর্জাতিক কোম্পানির অতি প্রিয় প্রশ্ন: "টেস্টবিহীন ৮ বছরের পুরনো কোডবেস পেলেন — কী করবেন?" মডেল উত্তর:

  1. আগে জাল বিছাই: বদলানোর আগে বর্তমান আচরণ ধরে রাখা characterization test লিখি — "এখন যা করে সেটাই সঠিক" ধরে নিয়ে ইনপুট-আউটপুট বন্দি করি।
  2. Seam খুঁজি: যেখানে interface বসানো যায় (DI-র জায়গা) — সেখান থেকে টুকরোটা টেস্টেবল করি।
  3. Strangler Fig প্যাটার্ন: পুরোটা একবারে rewrite নয় (প্রায় সবসময় ব্যর্থ হয়) — নতুন ফিচার নতুন পরিষ্কার মডিউলে, রাউটিং দিয়ে ধীরে ধীরে পুরনোটার রক্ত শুকিয়ে আনি।
  4. Boy-scout + মাপা অগ্রগতি: প্রতি sprint-এ coverage/complexity মেট্রিক একটু ভালো — বিগ-ব্যাং নয়, চক্রবৃদ্ধি।

Testing pyramid-ও এখানেই বলুন: অনেক unit (দ্রুত, সস্তা) → কম integration (১টা DB-সহ পথ) → হাতে-গোনা E2E (ভঙ্গুর, দামি)। উল্টো পিরামিড (সব E2E) = ধীর CI + flaky টেস্ট — চেনা রোগ।

Production Debugging, Troubleshooting ও Technical Documentation

"লোকালে তো ঠিকই চলে" — এর পরের অধ্যায়। জ্বলন্ত প্রোডাকশনে পদ্ধতিমাফিক তদন্ত, .NET-এর ডায়াগনস্টিক টুলবক্স, আর যে লেখালেখি আপনাকে টিমে অমূল্য করে তোলে।

৩৫.১ তদন্তের পদ্ধতি — আতঙ্ক নয়, প্রোটোকল

  1. Blast radius মাপুন: সবাই আক্রান্ত না ৫% ইউজার? এখনই খারাপ হলো না ধীরে ধীরে? (metrics ড্যাশবোর্ড — অধ্যায় ২৯)
  2. "কী বদলেছে?" জিজ্ঞেস করুন: প্রোডাকশন সমস্যার সিংহভাগের পেছনে সাম্প্রতিক deploy, config বদল, বা নির্ভরশীল সার্ভিসের বদল — deploy history-ই প্রথম সন্দেহভাজন।
  3. আগে থামান, পরে বুঝুন (mitigate first): rollback / feature flag বন্ধ / instance restart — ইউজারের রক্তক্ষরণ বন্ধ করুন; root cause খোঁজা তারপর, শান্ত মাথায়।
  4. Trace → লগ → কোড ক্রমে নামুন: ধীর/ব্যর্থ রিকোয়েস্টের trace খুলুন (কোন span-এ সময়/এরর?) → সেই সার্ভিসের structured লগ correlation ID ধরে ফিল্টার → তবেই কোডে।
  5. প্রতিটি অনুমান লিখে টেস্ট করুন: "মনে হয় ক্যাশ" — যাচাই না করে পরের অনুমানে লাফাবেন না; তদন্ত-নোট রাখুন (post-mortem-এও লাগবে)।

লক্ষ করুন runbook-এর চরিত্র: সিদ্ধান্ত-গাছ, প্রবন্ধ নয় — রাত ৩টার মানুষটা শুধু কপি-পেস্ট করবে। প্রতিটি recurring alert-এর একটা runbook — এটাই on-call সংস্কৃতির মেরুদণ্ড।

একই মানের দুই ইঞ্জিনিয়ারের মধ্যে যে লিখে বোঝাতে পারে — incident-এর post-mortem, ডিজাইনের ADR, পরিষ্কার PR বর্ণনা — promotion আর ভিসা-স্পনসর দুটোই তার দিকে ঝোঁকে। রিমোট/আন্তর্জাতিক টিমে লেখাই আপনার উপস্থিতি। আপনার GitHub প্রজেক্টে আজই একটা ADR ফোল্ডার খুলুন — ইন্টারভিউয়ারের চোখে এটা বিরল ও উজ্জ্বল সংকেত।

৩৫.৭ রিমোট আন্তর্জাতিক টিমের স্ট্যান্ডার্ড — যা দিয়ে আপনাকে মাপা হবে

রিমোট রোলে কোম্পানি আসলে দুটো জিনিস কিনছে: লিখে যোগাযোগ করার ক্ষমতা আর তদারকি ছাড়া দায়িত্ব নেওয়ার ক্ষমতা (ownership)। টেকনিক্যাল রাউন্ডের পাশাপাশি এগুলোই যাচাই হয়:

  • Async-first communication: "একটু কল করি?" নয় — প্রশ্নটা এমনভাবে লিখুন যেন উত্তরদাতা এক পাসেই দিতে পারে: কনটেক্সট + কী চেষ্টা করেছি + নির্দিষ্ট প্রশ্ন + স্ক্রিনশট/লিংক। এই দক্ষতার প্রদর্শনী হলো আপনার PR description, ticket comment, আর এই কোর্সের ৩৫.৬-এর সব দলিল।
  • Ownership-এর ভাষা: "ব্লকড ছিলাম, বসে ছিলাম" — নিষিদ্ধ বাক্য। বদলে: "X-এ ব্লকড, তাই Y অনুমান নিয়ে এগিয়েছি / Z-কে জিজ্ঞেস করে ততক্ষণে অন্য টাস্ক ধরেছি — কাল সকালের মধ্যে unblock না হলে escalate করব।" স্ট্যাটাস আপডেট চাওয়ার আগেই দিন — daily standup-এ কাজ + বাধা + পরের ধাপ, তিন লাইনে।
  • Timezone-সৌজন্য: overlap-ঘণ্টাগুলো (ইউরোপের সাথে বাংলাদেশের সুবিধা: দিনের বড় অংশ মেলে) মিটিং/pairing-এর জন্য রাখুন; বাকি সময়ের সিদ্ধান্ত লিখিত থাকুক যেন ঘুমন্ত সহকর্মী সকালে পড়ে নিতে পারে। "await async, block never" — টিমের ক্ষেত্রেও সত্য।
  • Behavioral প্রশ্নে রিমোট-সংস্করণ আসবে: "ভিন্ন timezone-এর সহকর্মীর সাথে দ্বিমত কীভাবে মেটাবেন?" (লিখিত proposal/ADR-এ যুক্তি, synchronous ঝগড়া নয়), "কীভাবে নিজেকে productive রাখেন?", "কখনো ভুল বুঝে ভুল জিনিস বানিয়েছেন?" (উত্তরের মোরাল: আগে লিখিত spec নিশ্চিত করি) — প্রতিটার STAR গল্প তৈরি রাখুন (২৪.৪)।
  • রিমোট ইন্টারভিউর যান্ত্রিক দিক: ইন্টারনেট ব্যাকআপ (মোবাইল হটস্পট রেডি), শান্ত জায়গা + ভালো মাইক, ক্যামেরা চালু, IDE/স্ক্রিন-শেয়ার আগে থেকে পরীক্ষিত। লাইভ কোডিংয়ে ইংরেজিতে think-aloud করুন — নীরবে কোড লেখা রিমোট-রোলে সবচেয়ে বড় red flag, কারণ ওরা দেখতে চায় আপনি দূর থেকে ভাবনা সম্প্রচার করতে পারেন কি না। আটকে গেলে বলুন: "I'd normally check the docs here — let me reason about it first."
  • Contractor হলে জানুন: অনেক আন্তর্জাতিক রিমোট চুক্তি B2B/contractor মডেলে হয় (Deel/Remote.com-এর মতো প্ল্যাটফর্মে) — invoice, নিজের ট্যাক্স — এই শব্দগুলো আগে শুনে রাখলে অফার-আলোচনায় হকচকিয়ে যাবেন না।
এক লাইনে

রিমোট ইন্টারভিউয়ের প্রতিটি ধাপে নিজেকে জিজ্ঞেস করুন: "আমার এই উত্তর/PR/মেসেজ কি এমন, যে ৮ ঘণ্টা পরে ঘুম থেকে উঠে পড়া সহকর্মীর আর কোনো প্রশ্ন করতে হবে না?" — হ্যাঁ হলে আপনি আন্তর্জাতিক স্ট্যান্ডার্ডে আছেন।

৩৫.৮ এই টপিকে সম্ভাব্য ইন্টারভিউ প্রশ্ন

  • প্রোডাকশনে API হঠাৎ ধীর — প্রথম ৫ মিনিটে কী করবেন? (blast radius → কী বদলেছে → mitigate — ৩৫.১)
  • CPU কম তবু সব টাইমআউট — কী সন্দেহ করবেন? (thread pool starvation — ৩৫.৩)
  • মেমোরি leak কীভাবে খুঁজবেন প্রোডাকশনে? (gcdump তুলনা — ৩৫.২)
  • Liveness আর readiness probe-এর পার্থক্য, গুলালে কী হয়? (৩৫.৩)
  • রাত ৩টায় alert — কোড দেখবেন আগে না rollback? (mitigate first — ৩৫.১)
  • "Tell me about a production incident you handled" (STAR + blameless post-mortem — ২৯.৩, ৩৫.৩)
  • নতুন সার্ভিসের কী কী ডকুমেন্ট রাখেন? ADR কী? (৩৫.৬)
  • Reproduce-না-হওয়া বাগ কীভাবে ধরবেন? (correlation ID + structured log + race সন্দেহ — ৩৫.৩)
  • ভিন্ন timezone-এর টিমমেটকে ব্লকার জানানোর আদর্শ উপায়? (async-first লিখিত এসকালেশন — ৩৫.৭)
